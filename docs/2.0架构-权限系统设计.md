# 2.0架构-权限系统设计

## 整体思路

由于该系统复杂度不高，所以基于RBAC进行设计，后续若文档模块需要更复杂的权限控制，可以再扩展ReBAC。

RBAC有以下几层抽象：用户、角色、权限

## 权限

可以对权限进行归类，可得到：

- 仅在前端使用的权限：页面组件的显示与隐藏、部分路由的访问
- 仅在后端使用的权限：对某个功能的分支处理，比如VIP用户可以使用AI提升翻译的代码质量
- 前后端通用的权限：缺少对笔记的访问权限时，前端应该隐藏对应的组件，后端应该返回无权限

于是做出以下抽象：

- 核心业务权限库：业务上是否具有某个权限
  - 不关心页面和具体接口，只关心业务上有没有这个能力
  - 该层权限的增删需要谨慎考虑，避免大量无关权限污染
- 映射层
  - 从核心业务权限库，派生出具体的权限给到前后端
  - 这一层支持定义简单的表达式，包括等于，与，或，非，比如：
    - 前端的笔记页面路由权限 = 核心业务权限库的笔记访问权限
    - 后端关于某模块的通知权限 = 核心业务权限库的通知使用权限 && 核心业务权限库的对应消息模块的订阅权限
- 执行权限库
  - 执行权限库分为前端库和后端库，代码中只能使用来自这里的权限码

权限码命名方式：以 “.” 分隔，推荐按照以下形式命名：“模块.对象.动作”

管理逻辑：

- 交互上，权限管理页面应该区分出三个表格分别对应三个权限库
- 由于执行权限库中有部分权限是映射来的，这部分权限则无法直接编辑有无授权，只能编辑映射关系
- 非映射的权限可以直接编辑有无授权

推荐的CI集成：

- 代码发版前，使用构建工具检查是否所有权限都已配置，如果没有则阻断发版

## 角色

按照RBAC架构，角色是权限的集合，角色可以拥有多个权限

## 用户

按照RBAC架构，用户是角色的集合，用户可以拥有多个角色，其权限为所有角色的权限的并集
